Artifact Description

Building on the software design foundation from Milestone Two, this enhancement addresses the performance layer. The original application performed every rescue-type filter as a full MongoDB collection scan — O(n) — repeated on every radio-button click. For the Austin Animal Center's dataset of hundreds of thousands of records, this is unacceptable at production scale.

The new animal_shelter_ds.py module introduces purpose-built in-memory data structures that replace repeated O(n) scans with sub-linear in-session lookups.

Data Structures Implemented

AnimalIndex — Hash Map O(1) avg lookup

An in-memory hash map keyed on animal identifiers. After loading records into memory, any individual record is retrieved in O(1) average time, eliminating repeated database round-trips for the same record within a session.

BreedInvertedIndex — Inverted Index O(k) lookup

Maps each normalized breed name to the set of animal IDs with that breed. Building the index is O(n); subsequent breed lookups are O(k) where k is the result-set size. The multi_lookup() method performs union across multiple breeds — the OR semantics used by all rescue-type filters — without touching non-matching records.

RescuePriorityQueue — Min-Heap O(log n) push/pop

Python's heapq min-heap with negated scores to simulate a max-heap, surfacing the highest-suitability animals first. top_n() uses heapq.nsmallest for O(k log n) retrieval without draining the queue.

Binary Search Age Filter O(n log n) sort + O(log n) query

Records are sorted by age once (O(n log n)), then lower-bound and upper-bound binary searches locate window boundaries in O(log n) each. Repeated queries reuse the sorted array, making each subsequent filter O(log n + k) rather than O(n).

Reflection

The most significant learning was the gap between knowing a data structure and knowing when to use it. The decision to use an inverted index required recognizing that breed filtering is essentially a set-membership query with OR semantics — exactly the problem inverted indexes are designed for. Recognizing that required understanding the access patterns of the application, not just the properties of the structure.

Implementing binary search without relying on Python's bisect module forced careful thinking about loop invariants. The distinction between a lower-bound search (first index >= target) and an upper-bound search (one past last index <= target) is subtle but critical for a half-open interval filter.
