Artifact Description

The artifact is the Grazioso Salvare Rescue Animal Dashboard, originally developed in CS 340: Client/Server Development during Spring 2025. The application connects a Python-based Dash web dashboard to a MongoDB database containing records from the Austin Animal Center (AAC), allowing staff to filter shelter animals by rescue suitability category and visualize results on an interactive map and breed chart.

The original implementation was a functional prototype: a single monolithic script with hardcoded credentials, no error handling, no logging, and no separation of concerns.

Justification & Skills Demonstrated

This artifact was selected because it presents a realistic, multi-layered software design challenge. The enhancement applied the single-responsibility principle by separating database access, application logic, security, and utility functions into distinct, independently testable modules. The result is a codebase that is maintainable, deployable across environments, and secure by design.

Files Introduced

  • .pyapp.pyRestructured Dash app, env-var config, Bootstrap layout
  • .pycrud.pyAnimalShelter class with typed CRUD + rescue query methods
  • .pycrud_operations.pyGeneric CRUD base class with projection, limit, aggregate
  • .pysecurity.pyInput sanitization, query validation, password hashing, HTTP headers
  • .pyutils.pyAnimal validation, formatting, rescue classification, statistics

Key Enhancements

  • Hardcoded credentials replaced with .env environment variables via python-dotenv
  • Comprehensive logging throughout all modules with structured timestamps
  • SecurityManager class: sanitizes input, rejects dangerous MongoDB operators ($where, $eval), hashes passwords with SHA-256 + salt
  • HTTP security headers: HSTS, X-Content-Type-Options, X-Frame-Options, CSP, X-XSS-Protection
  • Full type annotations (Optional[str], Dict[str, Any], etc.) throughout
  • Connection timeout (serverSelectionTimeoutMS=5000) prevents indefinite hangs

Reflection

The most significant learning was experiencing what "separation of concerns" means in practice beyond a textbook definition. When the original single-script application needed a security fix, there was no clean place to put it — everything was entangled. Building security.py as a dedicated module made it immediately clear which responsibilities belonged there and which belonged in crud.py or utils.py.

The challenge of deciding what constitutes a "utility" function versus a domain function also became concrete. The classify_rescue_type() function in utils.py knows about breeds and age ranges — domain knowledge — but is a pure function with no side effects. Placing it in utils.py rather than crud.py keeps the database layer focused on I/O, even though the function uses domain-specific constants.
