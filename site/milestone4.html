Artifact Description

The final enhancement addresses the database layer directly. The original application treated MongoDB as a passive repository with no index management, no schema validation, no connection pooling, and all data processing happening in Python. The new db_manager.py module changes all of this.

Database Enhancements

ConnectionPool — Explicit Pool Sizing

Wraps MongoClient with maxPoolSize=10, minPoolSize=2, and timeout settings. Exposes is_healthy() ping and reconnect() for graceful degradation. A context manager provides MongoDB client sessions for multi-document atomic operations.

IndexManager — COLLSCAN → IXSCAN

Creates four indexes: single-field on animal_type, sparse single-field on age_upon_outcome_in_weeks, a compound index on (animal_type, breed, age_upon_outcome_in_weeks) optimized for rescue queries, and a text index on name. The compound index converts O(n) collection scans to O(log n + k) index scans for all three rescue-type filters.

SchemaValidator — Write-Time Enforcement

Applies a JSON Schema using MongoDB's collMod command: animal_type must be one of a controlled vocabulary, breed must be a non-empty string, age must be between 0–2000 weeks, and coordinates must be valid geographic ranges. Uses "moderate" validation level to accommodate pre-existing non-conforming documents.

AggregationPipelines — Server-Side Computation

Four pipelines move computation into MongoDB: breed_distribution() groups by breed with an optional filter, age_statistics() computes min/max/avg/stddev, rescue_type_summary() uses $facet to evaluate all three rescue filters in a single round-trip, and outcome_type_distribution() groups by outcome type.

QueryPerformanceLogger — Observability

Wraps find() with explain() analysis that extracts the winning query plan stage, index name, documents examined vs. returned, and execution time. COLLSCAN operations are promoted to WARNING-level log entries, surfacing missing indexes immediately during development.

Reflection

The most significant learning was understanding where the boundary lies between application and database logic. The same computation that required loading all documents into a DataFrame and calling groupby() can instead be expressed as a two-stage pipeline that MongoDB executes using an index scan, returning only the aggregated result.

The tension between schema validation strictness and compatibility with legacy data was also a genuine challenge. Setting validationLevel="strict" immediately rejected updates to documents whose age_upon_outcome_in_weeks was stored as a BSON string rather than a number — a real data-quality issue in the AAC dataset. Switching to "moderate" resolved this and the trade-off is documented in the module's docstring.
